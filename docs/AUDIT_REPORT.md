# BAZ Marketing Hub — Deep Audit Report

> Generated by high-thinking-mode inspection of the entire `baz/` codebase.
> Covers architecture, content, UX, SEO, security, performance, and business logic.

---

## EXECUTIVE SUMMARY

BAZ is a **Next.js 14 + TypeScript + Tailwind** marketing agency site with an
embedded product ("Marketing Hub"), SQLite backend, AI integrations, lead
pipeline, and 18-service catalog. The codebase is well-structured with clear
separation of concerns. However, there are **critical blindspots** across
security, content authenticity, performance, and production readiness that
need to be addressed before public launch.

**Score: 7.2/10** — Strong architecture, significant gaps in production readiness.

---

## 🔴 CRITICAL BLINDSPOTS (Fix Before Launch)

### 1. All Content is Placeholder/Fictional
**Severity: CRITICAL**

- **All 12 case studies** use fictional company names (ViralVista, Northwind, EngageEra, etc.) with `placeholder` flags never set on CaseStudy type
- **All 8 testimonials** have `placeholder: true` — no real client quotes
- **All 6 team members** appear to be fictional personas (only Brahim ZERBOUT seems real)
- **All service proof points** use anonymized composites ("DTC Beauty · MENA", "B2B SaaS · US")
- **Blog posts** are authored by fictional personas (Sora Tanaka, Mira Okafor, Daniel K.)
- **Stats** are hidden in production (`null` when `NODE_ENV=production`) — good instinct, but means the hero trust strip is empty in prod

**Risk:** Publishing fabricated case studies and testimonials can violate FTC guidelines and damage credibility. The brand brief says "Numbers over adjectives" but the numbers aren't real.

**Action:** Replace with real client material, or clearly mark as "representative composites" with visible disclaimers. Remove `placeholder: true` flags only when real data is swapped in.

---

### 2. Default Owner Credentials Are Insecure
**Severity: CRITICAL**

```
OWNER_EMAIL=owner@baz.agency
OWNER_PASSWORD=changeme-on-first-login
```

This is in `.env.local` and hardcoded as fallback in `lib/db.ts`. If the env
isn't set, the app boots with a guessable owner account. The password is
bcrypt-hashed but the plaintext is in the codebase.

**Action:** 
- Force password change on first login
- Remove default password from code — require env var, fail boot if not set
- Add password complexity requirements

---

### 3. Admin Token Exposed in `.env.local`
**Severity: HIGH**

`ADMIN_TOKEN=38c30380f1d404820c36372967a7ca242eff1cbee49c95a9` is committed
in `.env.local`. While `.gitignore` should exclude it, the token is visible
in the working directory and could leak through backups or snapshots.

**Action:** Rotate the token. Ensure `.env.local` is in `.gitignore` (it is,
but verify in all snapshot dirs too).

---

### 4. No CSRF Protection on Server Actions
**Severity: HIGH**

`lib/actions.ts` exports `submitLead` as a server action with no CSRF token
verification. While Next.js server actions have built-in origin verification,
the `/api/leads` POST endpoint is fully public and has no CSRF protection.

**Action:** Add CSRF tokens to form submissions. Verify `Origin` header on
API POST routes.

---

### 5. Rate Limiter is In-Memory Only
**Severity: HIGH**

`lib/rate-limit.ts` uses an in-memory `Map`. In serverless/edge deployments,
each instance has its own map — meaning rate limits are per-instance, not
global. An attacker can bypass by hitting different instances.

**Action:** Swap with Redis/Upstash for production. The code already has a
comment acknowledging this.

## 🟠 HIGH PRIORITY IMPROVEMENTS

### 6. TypeScript Build Errors
**Severity: HIGH**

```
app/layout.tsx(108,11): error TS2322: Property 'attribute' does not exist on type 'IntrinsicAttributes & { children: ReactNode; }
```

The `ThemeProvider` (next-themes) props are not being recognized. Also 4 OG
route handlers have `contentType` type errors. The build currently succeeds
(because `next build` is more lenient than `tsc --noEmit`), but these are
real type issues.

**Action:** Fix the ThemeProvider typing. The issue is likely a version
mismatch between `next-themes` and its types. Add `contentType` to the OG
route exports or remove the property.

---

### 7. No Email Notification on Lead Capture
**Severity: HIGH**

Leads are written to SQLite and optionally forwarded to a webhook, but there's
no email notification to the team when a lead comes in. `RESEND_API_KEY` is
empty in `.env.local`. The contact form silently succeeds.

**Action:** Wire up Resend (or any email provider) to fire on lead creation.
The infrastructure is there (`NOTIFY_EMAIL`, `NOTIFY_FROM` env vars) but the
code to send doesn't exist.

---

### 8. No Real Analytics Integration
**Severity: MEDIUM-HIGH**

`NEXT_PUBLIC_GA4_ID` is empty. The Analytics component pushes to `dataLayer`
but no actual GA4, PostHog, or Mixpanel is configured. You're flying blind
on traffic, conversions, and user behavior.

**Action:** Set up GA4 or PostHog. The dataLayer infrastructure is ready —
just needs an ID.

### 9. Booking URL is Empty — CTAs Fall Back to mailto
**Severity: MEDIUM-HIGH**

`NEXT_PUBLIC_BOOKING_URL=` is empty. Every "Book a growth call" button
generates a `mailto:` link. This is a conversion killer — no calendar
scheduling, no automated reminders.

**Action:** Connect Cal.com, Calendly, or TidyCal. Set `NEXT_PUBLIC_BOOKING_URL`.

---

### 10. No Sitemap Submission or Search Console Verification
**Severity: MEDIUM**

The sitemap is generated dynamically (good!) but there's no Google Search
Console verification meta tag or DNS verification. No `google-site-verification`
in the layout.

**Action:** Add GSC verification. Submit sitemap. Add Bing Webmaster verification.

## 🟡 MEDIUM PRIORITY IMPROVEMENTS

### 11. Footer Links to Non-Existent Pages
**Severity: MEDIUM**

The footer links to `/marketing-hub/showcase`, `/triangle`, `/dive`,
`/attribution`, `/sequences`, `/nova`, `/cockpit` — these appear to be
Marketing Hub routes that live on a separate port (3010). Clicking them from
the main site will 404 unless the Marketing Hub is mounted at the same domain.

**Action:** Either:
- Make these links open `HUB_URL` in a new tab
- Or add catch-all routes that redirect to the Hub
- Or remove from footer if the Hub isn't publicly accessible

---

### 12. `dangerouslySetInnerHTML` in Marketing Hub Page
**Severity: MEDIUM**

```tsx
<p className="text-sm text-ink-600 mt-1" dangerouslySetInnerHTML={{ __html: s.d }} />
```

The triangle loop steps use `dangerouslySetInnerHTML` with static content
that includes `&apos;` entities. While the content is static (not user input),
this is unnecessary and a code smell.

**Action:** Replace with proper JSX. Use `{"'"}` or `&apos;` in JSX directly.

---

### 13. Dual Lead Submission Paths
**Severity: MEDIUM**

Both `lib/actions.ts` (`submitLead` server action) and `/api/leads` POST
route write leads to the same SQLite DB. While the comment says this "fixes
the dual-store bug," there are still two code paths with slightly different
logic:
- Server action: honeypot check, no scoring
- API route: rate limiting, deterministic scoring + LLM scoring

**Action:** Consolidate into a single `createLead()` function that both paths
call. Ensure both have: validation, honeypot, rate limiting, scoring, and
optional webhook forwarding.

---

### 14. No Image Optimization for Screenshots
**Severity: MEDIUM**

`noira-screen.png` (1.2MB), `noira-home.png` (817KB), and other PNGs in the
home directory are large. While these aren't in the baz/ project, the OG
images (`/og/default.png`, `/og/home-200k.png`) are served statically with
no `next/image` optimization.

**Action:** Use `next/image` for OG images where possible. Compress PNGs to
WebP. The OG route already generates SVG-based images dynamically (good).

---

### 15. Norobots.txt Disallows /book
**Severity: LOW-MEDIUM**

```
disallow: ['/api/', '/book']
```

`/book` is disallowed but there's no `/book` page in the app. The `/contact`
page (which should probably be disallowed or at least canonical'd) is
allowed. The `/admin/*` routes are not disallowed.

**Action:** Add `/admin/` to disallow list. Remove `/book` if it doesn't exist.
Add `/portal/` to disallow (it's a client-facing portal, not for indexing).

---

### 16. Inconsistent Dark Mode Implementation
**Severity: MEDIUM**

The site uses `next-themes` with `data-theme` attribute, but:
- The Tailwind config uses `darkMode: ['selector', '[data-theme=dark]']`
- CSS variables are swapped in `[data-theme='dark']` 
- Some components use `dark:` Tailwind variants (which require `class` strategy)
- The `dark:` variants won't work with `data-theme` selector strategy

**Action:** Audit all `dark:` class usage. Either:
- Switch to `class` dark mode strategy
- Or replace all `dark:` variants with CSS variable-based styling
- Or use `[data-theme=dark]` selector in CSS instead of `dark:` Tailwind utility

### 20. No Structured Data on Service Pages
Service pages have FAQs but no `FAQPage` JSON-LD schema being injected (the
`faqLd` function exists in `lib/seo.ts` but may not be called on service pages).

**Action:** Add `FAQPage` and `Service` schema to each `/services/[slug]` page.

### 21. No Compression or Caching Headers
The Next.js app doesn't set custom `Cache-Control` headers for static assets.
Vercel handles this automatically, but for self-hosted deployments, static
assets need long-cache headers.

**Action:** Add `headers()` export in `next.config.mjs` for static asset caching.

---

### 22. `package.json` Has Both `pnpm-lock.yaml` and `package-lock.json`
Two lockfiles exist. This can cause dependency resolution inconsistencies.

**Action:** Pick one package manager (npm or pnpm) and delete the other lockfile.

---

### 23. No Error Boundaries
There are no React Error Boundaries in the app. If a component throws during
render, the entire page crashes with a generic Next.js error.

**Action:** Add `error.tsx` files at the app root and key route segments.

---

### 24. No Loading States for Server Components
No `loading.tsx` files for routes that fetch data (services, case studies,
insights). The page is blank until the server finishes rendering.

**Action:** Add `loading.tsx` with skeleton states for data-heavy routes.

---

### 25. No Newsletter Signup Functionality
`NewsletterForm` component exists in marketing components but there's no API
endpoint or email provider integration to actually subscribe people.

**Action:** Wire up to Resend, Mailchimp, or ConvertKit. Add `/api/subscribe` route.

## 🏗️ ARCHITECTURE ASSESSMENT

### Strengths
1. **Clean separation**: content/, lib/, components/, app/ — well organized
2. **Type-safe**: TypeScript throughout, dedicated types in types/
3. **SEO foundation**: buildMetadata, JSON-LD, sitemap, robots, OG images
4. **Database**: SQLite with WAL mode, proper schema, audit log
5. **Auth**: Cookie-based sessions, bcrypt password hashing, proper token management
6. **AI integration**: Provider-agnostic LLM adapter (OpenAI, Anthropic, Ollama)
7. **Lead pipeline**: Scoring (deterministic + LLM), routing, audit trail
8. **Design system**: Comprehensive tokens, consistent components, brand brief
9. **Dynamic OG images**: Per-ICP hero variants with matching OG previews
10. **Responsive**: Mobile sheet menu, responsive grids, clamp() typography

### Weaknesses
1. **No test coverage** beyond Playwright config (no unit tests found)
2. **No CI/CD pipeline** visible (no GitHub Actions, no deploy script beyond `deploy.sh`)
3. **No environment validation** — app boots with insecure defaults
4. **No monitoring/observability** — `lib/observability.ts` exists but unclear usage
5. **No CDN or edge caching** strategy
6. **No A/B testing infrastructure** despite offering CRO services
7. **No content management** — all content is hardcoded in TypeScript files
